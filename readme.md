# Vertex Animation Texture in Godot

## VAT Concept
The concept of Vertex Animation Textures (VAT), a technique where vertex motion is baked into textures and played back on the GPU. Instead of using bones or CPU-driven deformation, each vertex reads its animated position (or offset) from a texture in the shader.
In this sample, the cube’s top face moves up and down based on data stored in the VAT. The accompanying texture visualizes this encoded animation, where pixel values represent vertex positions over time. This approach is highly efficient and commonly used for complex effects like simulations, crowds, or procedural deformations.

## Export from Blender
To export vertex animation textures from Blender using OpenVAT, ensure that all animations are combined into a single NLA strip—only animation data contained within an active NLA strip will be exported. Before exporting, push your actions to the NLA Editor and arrange them into one continuous strip if needed. For correct shading in Godot, vertex normals should be exported with the same texture on the lower half. 

For more detailed instructions on using the plugin, visit: https://openvat.org

> [!NOTE]
> If exportet as gltf the material is corrupted in Godot and can not be imported directly. Either export the VAT created Model seperatly or create a new Material.

## Import in Godot
### Overview
This repository contains **Vertex Animation Texture (VAT)** shaders for Godot `spatial` materials,for use with intance meshes.

It animates meshes entirely on the GPU using a texture generated by **OpenVAT**.

The generated texture contains:
- **Top half** → Vertex positions
- **Bottom half** → Vertex normals

Export requirements:
- Vertex Normals = Packed  
- Use Single Row = ON  
- Format = glTF 

## Controlling the OpenVAT Shader via Script (Godot)

This VAT shader is fully controllable from GDScript because all important animation parameters are exposed as **uniforms or per-instance data**.

You can control:
- animation playback
- per-instance timing
- frame range
- speed
- looping
- offsets
- pause/resume behavior



### Material-level control (global)
Affects **all instances / all meshes using this material**

Examples:
- `playing`
- `speed`
- `pause_frame`
- `vertex_animation_texture`
- `albedo`, `roughness`, etc.

### Instance-level control (MultiMesh or instance uniforms)

Affects **individual animated objects**

Examples:
- `instanceStartTime`
- `anim_phase_offset`
- `frame_start`
- `frame_end`

### Controlling via GDScript (Standard Mesh)

```gdscript
$MeshInstance3D.get_active_material(0).set_instance_shader_parameter()
```
---
This shader is designed as a flexible base system rather than a fixed solution. It is meant to be **modified and adapted** to fit your specific use case, whether that is crowds, VFX, vegetation, or animated characters.

You are encouraged to adjust:
- the frame logic
- instance data packing
- looping and timing behavior
- texture layout assumptions
- performance tradeoffs

The goal is not to enforce a single workflow, but to provide a **controllable VAT framework** that you can reshape to match your project’s needs without the use of a Multimesh.

### Notes
> [!NOTE]
> Shader time ≠ Time.get_ticks_msec() / 1000.0. It’s better to calculate time manually, preferably in a global scope.
```gdscript 
    var time:float;
    func _precess(delta)->void:
	    time += delta;
```
